## Context

A recent overhaul of user-level MCP servers and plugins broke three classes of references in `claude-code-toolkit`:

1. **`mcp__plugin_github_github__*`** (4 tools used by code-reviewer/tester/AGENT_TEAM/CLAUDE.local.md in all 6 variants) — the old standalone `github` MCP server is gone. Replacement: the `github` plugin (`@claude-plugins-official`) loads GitHub's hosted server at `https://api.githubcopilot.com/mcp/` and exposes the same tool *suffixes* under prefix `mcp__plugin_github_github__*`. Requires `GITHUB_PERSONAL_ACCESS_TOKEN` env var.

2. **Language/framework MCP servers** (`sqlite`, `dotnet-tools`, `rust-tools`, `windows-mcp`) — removed from user level. User's new policy: **register at project level only**, so the user-level context stays small and only loads them in repos that actually need them.

3. **Plugin migrations** (`context7`, `playwright`) — already correctly prefixed in templates (`mcp__plugin_context7_context7__*`); only docs and user-level-reference need cleanup.

Verified this session: the `github` plugin is loaded after setting `GITHUB_PERSONAL_ACCESS_TOKEN` and restarting. `ToolSearch` confirms all 4 required tools exist:
- `mcp__plugin_github_github__issue_read`
- `mcp__plugin_github_github__issue_write`
- `mcp__plugin_github_github__add_issue_comment`
- `mcp__plugin_github_github__pull_request_review_write`

## Decisions (user-locked)

1. **Project-level `.mcp.json` distribution**: `setup-project.{sh,ps1}` generates `.claude/.mcp.json` inline at setup time (option (b)). No `.mcp.json.template` file tracked per variant.
2. **Windows-MCP path**: setup script auto-configures it for `dotnet-maui` and `rust-tauri` variants.
3. **Scope**: all-in-one PR (github rename + layered MCP policy + doc updates + setup script changes + per-variant project-level `.mcp.json` generation).

## Open questions

1. Should the setup script's new `--mcp-dev-servers-path` flag have a sensible default (`~/repos/mcp-dev-servers` or `G:/git/mcp-dev-servers`)? Proposal: no default — if the flag is missing for a variant that needs it, print a warning and skip the affected entries. Users without `mcp-dev-servers` installed still get a working (minimal) setup.
2. Windows-MCP is currently unregistered at user level. Does "auto-configure" in the setup script mean we assume `uvx` is installed and on PATH? Proposal: yes. If the user doesn't have `uvx`, the MCP server entry will fail to load but won't break anything else. Document this in `mcp-servers/HOWTO.md` under the Windows-MCP subsection.
3. Should `--sqlite-db-path` accept a relative path (resolved against the target project directory) or require absolute? Proposal: accept both; `setup-project.sh` resolves relative paths against the project dir at write time.
